# 📂 文档组织结构说明

本文档说明项目文档的组织方式和分类原则。

## 📁 文档分类原则

### 1. architecture/ - 架构文档
**包含内容**: 项目架构设计、分层说明、模块划分等

**分类依据**: 
- 描述项目整体结构
- 说明设计理念和原则
- 不涉及具体实现细节

**文档示例**:
- `ARCHITECTURE.md` - 分层架构详细说明

### 2. api/ - API文档
**包含内容**: REST API接口文档、接口列表、使用示例等

**分类依据**:
- 面向外部调用者
- 描述接口规范和使用方法
- 不涉及内部实现

**文档示例**:
- `API.md` - 完整API接口文档

### 3. refactoring/ - 重构文档
**包含内容**: 重构过程记录、迁移步骤、清理报告等

**分类依据**:
- 记录历史重构过程
- 迁移和变更记录
- 一次性文档（重构完成后保留作为历史记录）

**文档示例**:
- `REFACTORING_GUIDE.md` - 重构指南
- `REFACTORING_COMPLETE.md` - 重构完成报告
- `CLEANUP_COMPLETE.md` - 清理完成报告
- 等等

### 4. guides/ - 使用指南
**包含内容**: 配置指南、使用说明、问题排查等

**分类依据**:
- 面向开发者和运维人员
- 提供操作指导
- 持续更新的文档

**文档示例**:
- `SECURITY.md` - 安全配置指南
- `LOGGING.md` - 日志配置指南
- `TROUBLESHOOTING.md` - 问题排查指南

## 📝 文档命名规范

### README.md
每个目录都有一个 `README.md` 作为该目录的索引和说明。

### 文档命名
- 使用大写字母开头的驼峰命名
- 使用描述性名称
- 统一使用 `.md` 扩展名

**示例**:
- `ARCHITECTURE.md`
- `REFACTORING_COMPLETE.md`
- `CLEANUP_COMPLETE.md`

## 🔄 文档更新策略

### 持续更新文档
- `architecture/` - 架构变化时更新
- `api/` - API变更时更新
- `guides/` - 配置或功能变化时更新

### 历史记录文档
- `refactoring/` - 重构完成后不再更新，保留作为历史记录

## 📚 新增文档建议

### 应该放在哪里？

- **架构相关** → `architecture/`
- **API相关** → `api/`
- **配置和使用** → `guides/`
- **变更和迁移记录** → `refactoring/`

### 文档格式建议

1. 使用Markdown格式
2. 添加清晰的标题和目录
3. 包含代码示例和截图（如需要）
4. 保持文档更新及时
